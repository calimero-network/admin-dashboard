Release Process

Semantic releases — merge to main triggers automatic versioning, release, and deployment

How Releases Work

1

Commit with Conventional Message

Use conventional commit format: feat: add feature (minor bump), fix: fix bug (patch bump), feat!: breaking change (major bump).

2

Open PR → Merge to Main

CI runs tests and linting. On merge to main, the release workflow triggers automatically.

3

Automatic Release

Semantic release analyzes commit messages, bumps version in package.json, creates a GitHub Release with auto-generated changelog, and builds the production bundle.

4

GitHub Pages Deploy

The built bundle is deployed to github.io/admin-dashboard as part of the release workflow.

Commit Message Format

feat: description
New feature → minor version bump (0.1.0 → 0.2.0). Triggers release.
fix: description
Bug fix → patch version bump (0.1.0 → 0.1.1). Triggers release.
feat!: description
Breaking change → major version bump (1.0.0 → 2.0.0). The ! marks the breaking change.
chore:, docs:, style:, refactor:
No release triggered. Use for non-user-facing changes.
BREAKING CHANGE: in footer
Alternate way to mark a breaking change — major bump regardless of commit type.

CI Workflow Steps

The release CI workflow (.github/workflows/) runs:

  1. Install dependencies — pnpm install
  2. Lint — ESLint + TypeScript type-check
  3. Test — unit test suite
  4. Build — pnpm build (Vite production build)
  5. Zip artifact — bundle the dist/ folder into a release archive
  6. Create GitHub Release — with changelog from commits since last release
  7. Deploy to GitHub Pages — publish dist/ to the gh-pages branch
